14. Incident Runbooks

Runbook: HIPAA Breach Response

  1. Identify scope: which tables, which patients, which time window. Query oe_audit_logs for anomalous access patterns.
  2. Isolate: if active exposure, immediately rotate SUPABASE_SERVICE_ROLE_KEY and CLERK_SECRET_KEY in Vercel env + Infisical. This forces all sessions to re-authenticate.
  3. Preserve evidence: export relevant oe_audit_logs rows before any DB changes.
  4. Count affected patients: if >500, federal reporting required within 60 days to HHS Office for Civil Rights.
  5. Patient notification: if <500, notify affected patients within 60 days. If >500, also notify prominent media in affected states.
  6. Document in /docs/breach-log/ with timestamp, scope, remediation steps.
  7. Notify Brad (NuStack) and practice owner (Dr. Alex) immediately regardless of scope.

Runbook: Clerk Auth Broken

  1. Check Vercel env: confirm CLERK_SECRET_KEY starts with sk_live_ (not sk_test_).
  2. Check Clerk dashboard: verify the application is active and keys match.
  3. Check Clerk status page for outages.
  4. If sk_test_ → sk_live_ rotation needed: update both CLERK_SECRET_KEY and NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY in Vercel env. Redeploy.

Runbook: Square Payment Failing

  1. Check Square dashboard for API status.
  2. Verify SQUARE_ACCESS_TOKEN and SQUARE_LOCATION_ID are set in Vercel env.
  3. Check /api/payments/square route for errors in Sentry.
  4. Verify Square environment (sandbox vs production) matches SQUARE_ENVIRONMENT env var.

22. Product Vision

The 3-year vision: AK Ultimate Dental runs on a single AI-powered platform that handles every non-clinical workflow autonomously — patient acquisition, onboarding, communication, billing, compliance, HR, and business intelligence — so Dr. Alex can focus entirely on dentistry. Core User Journeys

  1. New Patient Acquisition: Marketing site → appointment request → oe_leads created → AI drafts response → Approval queue → Staff approves → Lead converted to patient → Insurance verified → First appointment booked.

  3. Billing Flow: Appointment complete → Insurance claim → Eligibility check (Vyne when wired) → ERA posting → Patient balance → Square payment at chairside → Invoice generated.
  4. AI Phone Call: Patient calls → Vapi AI handles → Appointment booked or message taken → Transcript + summary stored in oe_calls → Staff reviews in Calls dashboard.
  5. Automated Recall: Monday 5pm → cron finds patients 6+ months overdue → AI drafts recall message → Approval queue → Staff approves or auto-sends → SMS/email sent via Twilio/Resend → Recall tracked in oe_patient_reactivation_sequences.
  6. No-Show Recovery: Every 30 min Mon-Sat 1-8pm → cron detects no-shows → Recovery SMS sent → Rebook tracked.

31. Change Management

Adding New Features

  1. Run /scope-check — is it needed? Right layer? Existing tool?
  2. If feature touches PHI: add HIPAA assessment (which tables, audit requirements, BAA impact).
  3. Max 3 files per micro-prompt. Name files at top. Build only those. Stop.
  4. Run npx tsc --noEmit after every pass. Fix errors before next pass.
  5. Commit after each passing micro-prompt: feat({scope}): {description}

Adding New Database Tables

  1. Name with oe_ prefix for patient/practice data.
  2. Always include: id (UUID), created_at (TIMESTAMPTZ), updated_at (TIMESTAMPTZ), deleted_at (TIMESTAMPTZ nullable) for soft delete.
  3. If table contains PHI: enable RLS immediately. Add to oe_audit_logs triggers for sensitive field reads.
  4. Apply via Supabase MCP execute_sql. No migration files — migrations are applied ad hoc.

Adding New Third-Party Integrations

  1. Confirm BAA status if integration will touch PHI.
  2. Store credentials in Infisical (primary) + Vercel env (fallback). Never commit.
  3. Add to secrets-contract section of this Bible.
  4. Add integration to integrations map in Section 9.
